Le vrai coût caché

Le PNG meurt en silence

  • Le code évolue, la carte reste figée
  • La mise à jour devient un coût caché, donc elle est abandonnée
  • On finit par accepter une dette documentaire comme une fatalité
Le problème c'est de pouvoir faire vivre ces connaissances dans le temps.

Pas la qualité du dessin.

Changer de paradigme

Trois principes pour une
architecture vivante

1. Une unique source de vérité Une sémantique complète, à même de représenter tout le système clairement. Lâcher le dessin, passer au diagramme.
2. Des perspectives multiples Contexte, structure, comportements, niveaux de détail : représenter un seul système sous différents angles
3. Un vrai cycle de vie Le développement logiciel a ses cycles : réconcilions l'architecture avec ces processus

Modélisation C4

C4 = un découpage simple de vos architectures

  • C1 — Contexte : avec quoi interagit avec ce système ?
  • C2 — Containers : comment le système fonctionne-t-il ?
  • C3 — Components : comment se compose un container ?
  • C4 — Code : quelles classes / quels modules implémentent les composants ?
  • Optionnel — Dynamique / Déploiement : comment ça s'exécute, où ça tourne ?
C4, c'est comme Google Maps :
un territoire unique,
plusieurs représentations, plusieurs niveaux de zoom,
en conservant la cohésion.

ADR-001 : digitaliser le stand café

ADR-001 Digital coffee.md 
# ADR-001 : le stand café numérique
## Status
Accepted

## Contexte
Le stand café du festival AleFest souffre de 3 problèmes :
- File d'attente physique : les festivaliers bloquent 15 min et ratent des concerts
- Pas de visibilité sur le temps d'attente
- Baristas débordés sans priorisation ni suivi des commandes

## Décision
Créer un système de prise de commande numérique au comptoir avec :
- Une borne/tablette pour saisir les commandes (staff ou self-service)
- Un écran barista affichant la file de préparation	
- Un backend simple gérant commandes, un autre backend les présentant aux baristas
- Une file de messages faisant communiquer les backends et permettant de découpler les traitements
- Une base de données pour persistence des états et  de la comptabilité

## Contraintes
- Les baristas doivent pouvoir marquer une commande "prête" en un tap
- Le festivalier attend au comptoir jusqu'à la remise de la boisson

## Conséquences
- Réduit la file physique au comptoir
- Donne une meilleure visibilité aux baristas sur les commandes en cours
- Pose une base simple pour une évolution mobile ultérieure

Partie 2

Evoluer & éprouver

Faire évoluer l'architecture et l'éprouver avec des scénarios

V1 · Le comptoir

Système local,
festivalier captif

  • Commande numérique au comptoir via Application Comptoir
  • Responsabilités des Service de Commandes et Service Préparation bien séparés
  • La file de message notifie seulement l'équipe de barista
  • Le festivalier attend au stand jusqu'à la remise de la boisson

V2 · Le café mobile

Système intégré,
festivalier libéré

  • S'intégrer à l'app du festival + Service Festival pour commander à distance
  • Notification Service pour se libérer de l'attente au comptoir
  • Profiter de la qualité de l'architecture v1, le socle événementiel est enrichi
  • Le café devient une expérience fluide, sans immobiliser le festivalier

V2 · live coding

Le modèle V2 : ce qui change vraiment par rapport à coffee-v1

coffee-v2/system-model.c4 · diff vs coffee-v1 
model {
  festivalSystem = System_External 'Système Festival' {
    festivalApp = Container_MobileApp 'Application Festival' { ... }
    festivalService = Container_Api 'Service Festival' { ... }
    festivalApp -[calls]-> festivalService 'API mobile festival' { technology 'HTTPS' }
  }
  alefestCoffee = System_New 'AleFest Café' {
    counterApp = Container_Spa 'Application Comptoir' { ... }
    baristaScreen = Container_Spa 'Écran Barista' { ... }
    orderService = Container_Api 'Service de Commandes' { ... }
    preparationService = Container_Api 'Service Préparation' { ... }
    notificationService = Container_Api 'Service Notification' { ... }
    eventBus = Container_Queue 'RabbitMQ' { ... }
    eventBus -[async]-> notificationService 'Livre order.preparing / order.ready' {
      technology 'AMQP'
    }
    notificationService -[uses]-> festivalSystem.festivalApp 'Envoie push notification' {
      technology 'Firebase Cloud Messaging'
    }
  }
  festivalier -[uses]-> festivalSystem.festivalApp 'Commande mobile et suivi'
  festivalier -[uses]-> alefestCoffee.counterApp 'Commande au comptoir (fallback)'
  barista -[uses]-> alefestCoffee.baristaScreen 'Prépare et remet la boisson'
  festivalSystem.festivalService -[uses]-> alefestCoffee.orderService 'Intégration des commandes partenaire' { technology 'HTTPS' }
}

Que retenir et comment repartir avec du concret

Le paradoxe réconcilié : avant / après

Sujet Avant · le dessing Après · le diagram as code
Source de vérité Diluée dans Visio, Confluence et le souvenir du dernier atelier Le modèle .c4 dans Git
Mise à jour Nouveau fichier, nouveau dessin Modifier et capitaliser sur l'existant
Git / PR Le diagramme est sauvegardé quelque part Suivi de version
Validation Ambiguité sémantique, manque de cohérence Vue générée, diff lisible, review partagée
CI/CD Bouton d'export Validation et publication intégrées, comme une livraison de code!
Prise en main Difficile de produire un schéma fiable et à jour Mon LLM fait presque tout

  • Intégration de modèles entre eux
  • Export : d2, plantuml, mermaid, png...
  • Le cockpit de votre système, pas un screenshot figé
